# ADR-005: Adopt Sentence Transformer Embeddings

## Status

Accepted

---

## Context

The Libris relies on semantic embeddings to transform textual knowledge into a numerical representation suitable for similarity search.

Embeddings form the foundation of the Knowledge Index and directly influence retrieval quality.

The selected embedding model should provide:

- Strong semantic understanding
- Deterministic output
- Local execution
- Reasonable computational requirements
- Compatibility with academic textbook content

The embedding model should remain an infrastructure concern rather than a domain dependency.

---

## Decision

The platform shall adopt **Sentence Transformers** as the default embedding framework.

A pre-trained Sentence Transformer model will generate embeddings for:

- Document chunks
- User queries

The same embedding model must always be used for both indexing and retrieval.

Changing the embedding model shall require rebuilding the Knowledge Index.

The Embedding Engine shall expose a stable abstraction independent of any specific model implementation.

---

## Rationale

### Semantic Retrieval

Sentence Transformers are specifically designed to represent semantic similarity rather than lexical similarity.

This enables users to ask natural-language questions without relying on exact textbook wording.

---

### Local Execution

Models execute entirely on the user's machine.

This aligns with the Local-First philosophy adopted by the platform.

---

### Deterministic Output

Given identical:

- input text
- model
- configuration

the generated embedding remains identical.

This property is essential for reproducibility and evaluation.

---

### Mature Ecosystem

Sentence Transformers are well-supported, actively maintained, and widely adopted within the information retrieval community.

This reduces implementation risk.

---

### Educational Value

The framework provides an accessible introduction to modern semantic retrieval while remaining representative of production systems.

Students gain practical exposure to embedding-based retrieval without unnecessary complexity.

---

## Consequences

### Positive

- High semantic retrieval quality
- Local execution
- Deterministic embeddings
- Strong community support
- Easy experimentation with alternative models
- Good balance between quality and computational cost

### Negative

- Embedding generation can be computationally expensive for large document collections
- Re-indexing is required when changing embedding models
- Larger models increase storage requirements and indexing time

These trade-offs are acceptable for the intended workload.

---

## Alternatives Considered

### OpenAI Embeddings

Rejected.

Requires internet connectivity and external API usage, conflicting with the platform's Local-First philosophy.

---

### Gemini Embeddings

Rejected.

Introduces cloud dependency and external service requirements.

---

### Ollama Embeddings

Deferred.

Potential future option, but Sentence Transformers provide a more mature and broadly adopted retrieval ecosystem for the MVP.

---

### Traditional TF-IDF

Rejected.

Provides lexical similarity rather than semantic understanding.

Unable to effectively capture conceptual similarity expressed using different terminology.

---

### BM25 Only

Rejected.

Keyword retrieval alone is insufficient for natural-language academic questions.

Future versions may combine BM25 with semantic retrieval through hybrid retrieval.

---

## Implications

The Embedding Engine owns all embedding generation.

No other subsystem may generate embeddings independently.

All semantic search operations must assume a consistent embedding space.

The Retrieval Engine shall remain independent of the specific embedding framework.

Embedding models should be configurable through application configuration rather than hard-coded into the application.

## Notes

The platform adopts the Sentence Transformers framework rather than a specific embedding model.

Model selection is considered a configuration decision rather than an architectural decision.

This allows experimentation with improved embedding models while preserving the architecture and maintaining compatibility with the Embedding Engine abstraction.
