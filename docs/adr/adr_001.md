# ADR-001: Adopt Clean Architecture

## Status

Accepted

---

## Context

The Libris is intended to evolve beyond a simple semester project into a reusable academic Knowledge Retrieval Platform.

The platform contains multiple independent responsibilities, including:

- Document ingestion
- Text processing
- Knowledge indexing
- Retrieval
- Grounding
- Response generation
- Citation generation
- Presentation

Without architectural boundaries, these responsibilities would gradually become tightly coupled, making the codebase increasingly difficult to understand, extend, and test.

The project is also expected to evolve over time by introducing new retrieval techniques, embedding models, language models, and storage providers. These changes should not require significant modifications to business logic.

An architecture is therefore required that separates business rules from implementation details.

---

## Decision

The project shall adopt **Clean Architecture** as its primary architectural style.

The system will be organized into independent layers with inward-facing dependencies.

```
Presentation Layer
        │
Application Layer
        │
Domain Layer
        │
Infrastructure Layer
```

The Domain Layer shall remain independent of frameworks, external services, databases, and user interfaces.

Infrastructure components shall implement abstractions defined by higher-level layers rather than the reverse.

---

## Rationale

Clean Architecture provides several advantages aligned with the project's long-term objectives.

### Separation of Concerns

Each architectural layer owns a clearly defined responsibility.

Business rules remain independent of infrastructure.

---

### Replaceability

Infrastructure components can be replaced with minimal impact on the rest of the system.

Examples include:

- changing the vector database
- changing the embedding model
- changing the language model
- changing the PDF parser

---

### Testability

Business logic can be tested independently of external dependencies.

Individual Engines can be verified without requiring a running database or language model.

---

### Maintainability

A modular architecture reduces coupling and encourages incremental development.

New features can be introduced without affecting unrelated components.

---

### Scalability

The architecture supports gradual expansion into a larger academic Knowledge Retrieval Platform.

Additional capabilities should integrate through existing abstractions rather than modifying core business logic.

---

## Consequences

### Positive

- Clear separation of responsibilities
- Independent Engine development
- Easier unit testing
- Easier integration testing
- Improved maintainability
- Framework independence
- Infrastructure replaceability
- Better documentation alignment

### Negative

- Increased number of files
- Higher initial design effort
- More abstractions than a simple prototype
- Slightly steeper learning curve

These trade-offs are considered acceptable given the project's expected evolution.

---

## Alternatives Considered

### Layered MVC

Rejected.

MVC tightly couples application logic with presentation concerns and provides limited separation between domain logic and infrastructure.

---

### Monolithic Service Architecture

Rejected.

A single service layer would become increasingly difficult to maintain as additional Engines are introduced.

---

### Microservices

Rejected.

The project is intentionally designed as a local-first application.

Distributed architecture would introduce unnecessary complexity without providing proportional benefits.

---

### Framework-Centric Architecture

Rejected.

Allowing framework conventions to dictate project structure would reduce portability and make architectural evolution more difficult.

---

## Implications

All future implementation should respect the following dependency rule:

Higher-level policies must never depend directly on lower-level implementation details.

The Domain Layer defines interfaces.

The Infrastructure Layer implements them.

Presentation communicates only through the Application Layer.

## Notes

Clean Architecture is adopted as an architectural principle rather than a strict framework implementation.

The objective is to preserve clear dependency boundaries, independent business logic, and long-term maintainability while allowing pragmatic implementation decisions where appropriate.
