# ADR-009: Adopt Evaluation-Driven Development

## Status

Accepted

---

## Context

The primary objective of Libris is not simply to generate responses, but to generate responses that are trustworthy, grounded, and verifiable.

Unlike traditional software systems, retrieval-based AI systems cannot be evaluated solely through functional testing.

A system may:

- compile successfully
- pass all unit tests
- pass all integration tests

while still retrieving irrelevant evidence or generating poor academic responses.

The platform therefore requires an engineering approach in which retrieval quality is continuously measured throughout development.

Evaluation should become an integral part of the development lifecycle rather than a post-development activity.

---

## Decision

The platform shall adopt **Evaluation-Driven Development (EDD)**.

Every significant architectural change affecting retrieval or generation shall be evaluated using predefined quality metrics before being considered complete.

Evaluation shall complement, rather than replace, traditional software testing.

The Evaluation Framework shall define standardized benchmarks, datasets, metrics, and reporting procedures.

Development decisions should be informed by measurable improvements rather than subjective observations.

---

## Rationale

### Objective Decision Making

Architectural improvements should be supported by measurable evidence.

Changes should demonstrate improved retrieval quality, citation accuracy, or response groundedness.

---

### Continuous Improvement

Evaluation enables incremental optimization throughout development.

Each architectural iteration should produce measurable progress rather than relying on anecdotal observations.

---

### Regression Prevention

Evaluation metrics establish performance baselines.

Future modifications can be compared against previous results to detect unintended quality regressions.

---

### Educational Value

One objective of this project is to understand Retrieval-Augmented Generation systems beyond implementation.

Measuring retrieval quality provides practical experience with Information Retrieval evaluation methodologies commonly used in research and production systems.

---

### Reproducibility

Evaluation encourages reproducible experimentation.

Architectural decisions become supported by quantitative evidence rather than subjective preference.

---

## Evaluation Lifecycle

```
Implement

↓

Verify

↓

Evaluate

↓

Compare

↓

Improve

↓

Repeat
```

Testing verifies correctness.

Evaluation measures quality.

---

## Evaluation Scope

Evaluation should measure multiple aspects of the platform.

Examples include:

### Retrieval

- Recall@K
- Precision@K
- Mean Reciprocal Rank
- Context Coverage

---

### Generation

- Groundedness
- Faithfulness
- Completeness
- Clarity

---

### Citation

- Citation Accuracy
- Source Traceability
- Metadata Correctness

---

### Performance

- Indexing Time
- Retrieval Latency
- Response Time

---

### Reliability

- Hallucination Rate
- Determinism
- Reproducibility

---

## Consequences

### Positive

- Data-driven engineering decisions
- Improved retrieval quality
- Easier benchmarking
- Better regression detection
- Stronger academic credibility
- Continuous quality improvement

### Negative

- Additional implementation effort
- Benchmark maintenance required
- Evaluation datasets require ongoing curation
- Longer development cycle

These trade-offs are justified by the resulting increase in platform quality and trustworthiness.

---

## Alternatives Considered

### Manual Testing Only

Rejected.

Subjective observations cannot reliably measure retrieval quality or detect regressions.

---

### Evaluation After Completion

Rejected.

Delaying evaluation until the end of development increases the cost of correcting architectural deficiencies.

---

### User Feedback Only

Rejected.

While valuable, user feedback should complement objective evaluation metrics rather than replace them.

---

## Implications

Major modifications involving:

- Retrieval strategies
- Embedding models
- Prompt construction
- Language models
- Knowledge Index
- Context assembly

should be evaluated before adoption.

Architectural decisions should reference evaluation results whenever possible.

Benchmark reports should become part of the project's engineering documentation.

## Notes

Evaluation-Driven Development extends traditional software engineering practices into the domain of Information Retrieval and AI systems.

The objective is not merely to build a functioning Libris, but to continuously measure and improve its ability to retrieve trustworthy academic evidence and generate grounded, verifiable responses.

Features are considered successful only when they demonstrate measurable improvements against established evaluation criteria.
