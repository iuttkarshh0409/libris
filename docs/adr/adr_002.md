# ADR-002: Adopt Feature-First Project Structure

## Status

Accepted

---

## Context

The Libris consists of multiple independent capabilities that evolve at different rates.

Examples include:

- Document Management
- Knowledge Ingestion
- Retrieval
- Grounding
- Response Generation
- Citation Generation
- Configuration
- Evaluation

A traditional layer-first project structure organizes source code by technical concern.

Examples include:

```
controllers/

services/

repositories/

models/

utils/
```

While suitable for small applications, this organization becomes increasingly difficult to navigate as the project grows.

Related functionality becomes scattered across multiple directories, making maintenance, onboarding, and feature evolution more difficult.

The project is expected to evolve continuously throughout multiple academic semesters and should therefore prioritize long-term maintainability.

---

## Decision

The project shall adopt a **Feature-First Project Structure**.

The codebase will be organized around business capabilities rather than implementation technologies.

Each feature owns its complete implementation while respecting the dependency boundaries established by Clean Architecture.

A representative structure is shown below.

```
backend/

src/

    application/

    domain/

    infrastructure/

    presentation/

    shared/

frontend/

src/

    features/

        library/

        query/

        evidence/

        dashboard/

        settings/

    components/

    layouts/

    shared/
```

Features should encapsulate their own responsibilities while sharing only well-defined abstractions.

---

## Rationale

### Improved Discoverability

Developers should be able to locate all code related to a business capability without navigating multiple unrelated directories.

---

### Better Cohesion

Files that change together should remain together.

Document-related functionality should exist within the Document feature rather than being distributed across controllers, services, repositories, and models.

---

### Easier Maintenance

Feature boundaries reduce unintended coupling.

Changes to one capability should have minimal impact on unrelated areas of the system.

---

### Architectural Alignment

Feature-first organization naturally complements Clean Architecture.

Each feature contains its own application, domain, infrastructure, and presentation responsibilities while respecting dependency rules.

---

### Scalability

New capabilities can be introduced without restructuring the existing project.

Future features such as:

- Flashcards
- Study Sessions
- Quiz Mode
- Diagram Retrieval

can be added as independent modules.

---

## Consequences

### Positive

- Improved project navigation
- Higher feature cohesion
- Lower coupling
- Easier onboarding
- Better scalability
- Cleaner ownership boundaries
- Simpler incremental development

### Negative

- Slight duplication of directory structures
- More folders compared to small projects
- Requires discipline to maintain boundaries

These trade-offs are acceptable in exchange for long-term maintainability.

---

## Alternatives Considered

### Layer-First Organization

Rejected.

Related functionality becomes fragmented across technical layers.

Understanding a single feature requires navigating multiple directories.

---

### Flat Project Structure

Rejected.

Scales poorly beyond small prototypes.

Encourages tightly coupled code.

---

### Package-by-Layer

Rejected.

Prioritizes implementation technology over business capability.

This conflicts with the project's domain-oriented architecture.

---

## Implications

New functionality should be introduced as features rather than as additions to generic utility folders.

Shared components should exist only when they genuinely represent reusable abstractions.

Utility directories should remain intentionally small.

Business logic should never migrate into generic helper modules simply for convenience.

## Notes

Feature-first organization is adopted as a code organization strategy rather than an architectural style.

Architectural boundaries remain governed by Clean Architecture.

Feature boundaries improve discoverability, maintainability, and long-term evolution without altering dependency direction.
