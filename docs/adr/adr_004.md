# ADR-004: Adopt ChromaDB as the Initial Knowledge Index

## Status

Accepted

---

## Context

The Libris requires a vector database capable of storing semantic embeddings together with rich metadata for efficient similarity search.

The Knowledge Index is a foundational infrastructure component supporting:

- Embedding persistence
- Semantic retrieval
- Metadata filtering
- Citation traceability
- Knowledge Index rebuilding

The selected vector database should support the objectives of the MVP while remaining replaceable through architectural abstractions.

---

## Decision

The platform shall adopt **ChromaDB** as the initial implementation of the Knowledge Index.

ChromaDB will be used to:

- Store document embeddings
- Store chunk metadata
- Perform semantic similarity search
- Support local persistence
- Support deterministic index rebuilding

The application shall access ChromaDB only through repository abstractions defined by the Domain Layer.

No business logic shall depend directly on ChromaDB APIs.

---

## Rationale

### Local-First Compatibility

ChromaDB operates entirely on the local machine.

It aligns naturally with the platform's Local-First architecture.

---

### Lightweight Deployment

No dedicated database server is required.

This simplifies:

- Installation
- Development
- Testing
- Distribution

---

### Metadata Support

Each embedding can store associated metadata including:

- Book
- Chapter
- Section
- Page
- Chunk Identifier

Metadata is essential for retrieval transparency and citation generation.

---

### Python Ecosystem

ChromaDB integrates well with the intended technology stack while requiring minimal infrastructure configuration.

---

### Educational Value

The platform is intended not only as a study tool but also as a learning project.

Using ChromaDB provides practical exposure to vector databases without introducing unnecessary operational complexity.

---

## Consequences

### Positive

- Local persistence
- Simple setup
- Good metadata support
- Fast semantic retrieval
- Easy integration
- Suitable for educational workloads
- Minimal operational overhead

### Negative

- Limited distributed capabilities
- Not optimized for enterprise-scale deployments
- Future migration may require data conversion
- Advanced indexing strategies may be unavailable

These limitations are acceptable for the current scope of the project.

---

## Alternatives Considered

### FAISS

Rejected.

Provides excellent vector search performance but lacks built-in metadata management and persistence capabilities expected by the platform.

---

### Qdrant

Deferred.

Provides advanced production capabilities but introduces additional operational complexity beyond the needs of the MVP.

---

### Weaviate

Rejected.

Cloud-oriented architecture and infrastructure requirements conflict with the Local-First philosophy.

---

### Milvus

Rejected.

Powerful but unnecessarily complex for a single-user academic Knowledge Retrieval Platform.

---

### Pinecone

Rejected.

Cloud dependency conflicts with the platform's default Local-First operating model.

---

## Implications

The Indexing Engine shall interact with the Knowledge Index through an abstraction layer.

The Retrieval Engine should remain unaware of the underlying vector database implementation.

Future migration to another vector database should require changes only within the Infrastructure Layer.

No component outside Infrastructure may import ChromaDB directly.

## Notes

ChromaDB is selected as the initial implementation of the Knowledge Index rather than as a permanent architectural dependency.

The platform architecture intentionally isolates vector database implementations behind repository interfaces, allowing future adoption of alternative technologies without affecting business logic.
