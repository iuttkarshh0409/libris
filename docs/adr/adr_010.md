# ADR-010: Adopt a Deterministic Knowledge Processing Pipeline

## Status

Accepted

---

## Context

The Libris transforms academic textbooks into searchable semantic knowledge through a multi-stage processing pipeline.

The pipeline consists of multiple independent Engines responsible for:

- Document Parsing
- Text Processing
- Chunk Generation
- Embedding Generation
- Knowledge Index Construction
- Semantic Retrieval
- Prompt Construction
- Response Generation
- Citation Generation

Without deterministic processing, identical documents may produce different chunks, embeddings, retrieval results, or responses across executions.

Such inconsistencies make debugging, testing, benchmarking, and evaluation significantly more difficult.

Since the platform is intended for academic reference rather than creative content generation, reproducibility is considered a fundamental architectural requirement.

---

## Decision

The platform shall adopt a **Deterministic Knowledge Processing Pipeline**.

Whenever practical, identical inputs under identical configuration shall produce identical outputs.

Determinism shall apply to every Engine responsible for transforming knowledge.

Where complete determinism cannot be guaranteed (for example, during language generation), the platform shall minimize variability through controlled configuration and explicit documentation.

---

## Rationale

### Reproducibility

Academic tools should produce consistent results.

Students asking the same question against the same textbooks should receive equivalent evidence and citations.

---

### Debugging

Deterministic behavior makes failures reproducible.

Unexpected behavior can be investigated without uncertainty introduced by changing intermediate results.

---

### Evaluation

Meaningful benchmarking requires stable outputs.

Evaluation metrics lose significance if retrieval behavior changes unpredictably between executions.

---

### Testing

Deterministic components simplify:

- Unit Testing
- Integration Testing
- Regression Testing

Expected outputs remain stable over time.

---

### Trust

Users should develop confidence that the platform behaves consistently rather than unpredictably.

Consistency reinforces credibility.

---

## Pipeline Determinism

Each Engine should strive for deterministic behavior.

### Document Engine

Given the same document:

- Metadata extraction
- Page detection
- Parsing

should remain identical.

---

### Chunking Engine

Given identical configuration:

- Chunk boundaries
- Chunk identifiers
- Chunk ordering

should remain identical.

---

### Embedding Engine

Given:

- identical model
- identical configuration
- identical input

embeddings should remain identical.

---

### Indexing Engine

Knowledge Index construction should remain reproducible.

Rebuilding an index without changing configuration should produce equivalent searchable content.

---

### Retrieval Engine

Given:

- identical query
- identical Knowledge Index
- identical retrieval configuration

the Retrieval Context should remain identical.

---

### Grounding Engine

Prompt construction should be fully deterministic.

Prompt objects should contain identical sections, ordering, and metadata for identical inputs.

---

### Generation Engine

Generation may introduce controlled variability depending on the selected language model.

Where possible, configuration should minimize randomness.

Provider-specific limitations should be documented.

---

### Citation Engine

Citation generation should remain fully deterministic.

Identical evidence should always produce identical citations.

---

## Consequences

### Positive

- Improved reproducibility
- Reliable evaluation
- Easier debugging
- Stronger regression testing
- Greater user trust
- Consistent platform behavior

### Negative

- Some implementation choices may prioritize stability over flexibility.
- Certain language model providers may not support complete determinism.
- Configuration management becomes more important.

These trade-offs are acceptable given the platform's educational purpose.

---

## Alternatives Considered

### Fully Non-Deterministic Processing

Rejected.

Would undermine reproducibility and make evaluation unreliable.

---

### Deterministic Retrieval Only

Rejected.

Pipeline consistency requires deterministic behavior across all processing stages wherever practical.

---

### Randomized Retrieval Strategies

Rejected.

Randomized retrieval complicates debugging and reduces confidence in evaluation metrics.

---

## Implications

Configuration changes affecting:

- chunk size
- overlap
- embedding model
- retrieval parameters
- prompt structure

should be treated as architectural changes.

Such modifications may invalidate previous benchmarks and require re-indexing or re-evaluation.

Evaluation reports should always record the configuration used during execution.

## Notes

Determinism is adopted as a guiding engineering principle rather than an absolute guarantee.

The platform recognizes that some external language model providers may introduce unavoidable variability.

Where determinism cannot be enforced, variability should be minimized, documented, and isolated from the remainder of the Knowledge Processing Pipeline.

The ultimate objective is to ensure that the platform behaves predictably, transparently, and reproducibly while preserving the flexibility required to integrate modern language models.
