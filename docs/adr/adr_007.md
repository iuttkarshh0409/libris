# ADR-007: Adopt Engine-Based Domain Architecture

## Status

Accepted

---

## Context

The Libris consists of multiple independent capabilities that collectively transform academic documents into grounded, verifiable responses.

These capabilities include:

- Document ingestion
- Semantic chunking
- Embedding generation
- Knowledge indexing
- Semantic retrieval
- Prompt construction
- Response generation
- Citation generation

A traditional service-oriented architecture often centralizes multiple responsibilities into large service classes.

Examples include:

- RetrievalService
- RAGService
- AIService
- KnowledgeService

As application complexity increases, these services tend to accumulate unrelated responsibilities, resulting in tightly coupled, difficult-to-maintain components.

The platform requires an architectural model that emphasizes clear ownership, explicit boundaries, and independent evolution.

---

## Decision

The platform shall adopt an **Engine-Based Domain Architecture**.

An Engine represents a self-contained domain capability responsible for transforming one well-defined input into one well-defined output.

Each Engine owns:

- Its domain models
- Its application logic
- Its infrastructure adapters
- Its testing
- Its documentation

Every Engine shall expose a clear public interface while hiding internal implementation details.

## Rationale

### Single Responsibility

Every Engine performs one clearly defined capability.

Responsibilities should never overlap.

---

### High Cohesion

Related logic remains within the same Engine.

Functionality that changes together should remain together.

---

### Low Coupling

Engines communicate through domain objects rather than implementation details.

No Engine should depend on another Engine's internal implementation.

---

### Independent Evolution

Each Engine can evolve independently.

Changes within one Engine should have minimal impact on others.

---

### Improved Testability

Every Engine can be:

- unit tested
- integration tested
- benchmarked
- evaluated

independently.

---

### Improved Observability

Engine boundaries naturally support:

- logging
- metrics
- tracing
- debugging

without introducing cross-cutting complexity.

---

## Engine Communication

Engines communicate exclusively through typed domain objects.

Example

```
Book

↓

Document Engine

↓

Document

↓

Chunking Engine

↓

Knowledge Chunks

↓

Embedding Engine

↓

Embeddings

↓

Indexing Engine

↓

Knowledge Index

↓

Retrieval Engine

↓

Retrieval Context

↓

Grounding Engine

↓

Prompt

↓

Generation Engine

↓

Raw Response

↓

Citation Engine

↓

Final Response
```

Each Engine consumes one well-defined object and produces another.

---

## Consequences

### Positive

- Strong ownership boundaries
- High cohesion
- Low coupling
- Independent testing
- Independent benchmarking
- Clear architectural responsibilities
- Easier onboarding
- Better scalability

### Negative

- Increased number of components
- More explicit interfaces
- Slightly higher architectural complexity

These trade-offs are justified by improved maintainability and long-term extensibility.

---

## Alternatives Considered

### Large Service Classes

Rejected.

Services accumulate multiple unrelated responsibilities over time.

---

### Generic Pipeline Stages

Rejected.

Pipeline stages describe execution order rather than domain ownership.

---

### Layer-Only Architecture

Rejected.

Layers define dependency direction but do not define business capabilities.

---

### Microservices

Rejected.

Distributed deployment is unnecessary for the intended scope.

The desired modularity can be achieved within a modular monolithic architecture.

---

## Implications

Every new capability introduced into the platform should be evaluated to determine whether it belongs within an existing Engine or requires a new Engine.

Engines should never become generic containers for unrelated logic.

Business rules belong inside Engines.

Infrastructure remains outside the Engine's domain model.

## Notes

An Engine represents a business capability rather than an implementation pattern.

The objective is to model the platform around meaningful domain responsibilities instead of technical utilities or framework conventions.

This architecture intentionally favors explicit ownership, predictable data flow, and independently evolvable capabilities.
