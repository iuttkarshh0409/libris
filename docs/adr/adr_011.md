# ADR-011: Decouple Prompt Serialization from Domain Entity

## Status

Accepted

---

## Context

The Libris utilizes structured prompts to ground language models in verified textbook evidence before generating answers.

The `Prompt` is a core domain entity representing the logical request made to the generation stage. Attributes include:

- Query ID
- Retrieval Context ID
- System Instructions
- Task Definition

Currently, the `Prompt` entity also includes a `raw_prompt_string` attribute representing the compiled, raw text string ready for inference.

However, different language models and model providers (e.g., Google Gemini, Anthropic Claude, local Llama instances) require different prompt structures, system instruction placements, and formatting tags (such as chat templates, XML delimiters, or role markers).

Embedding the compiled `raw_prompt_string` within the domain `Prompt` entity couples the Domain Layer to the serialization format of specific infrastructure providers, violating Clean Architecture boundaries.

---

## Decision

The platform shall **decouple prompt serialization from the Prompt domain entity**.

Specifically:

1. The `Prompt` domain entity shall only represent the logical components of the prompt (System Instructions, Task Definition, Retrieval Context, Query Reference, and Metadata). It shall NOT contain a serialized `raw_prompt_string` field.
2. A new interface `PromptCompiler` (or `PromptSerializer`) shall be defined. This interface will belong to the Domain Layer (e.g. within `src/domain/providers/` or `src/domain/engines/`).
3. Concrete implementations of `PromptCompiler` shall reside in the Infrastructure Layer, typically associated with concrete Language Model providers (e.g., `GeminiPromptCompiler` or `LlamaPromptCompiler`).
4. The `PromptCompiler` will transform the logical `Prompt` domain entity into the specific format required by the target Language Model provider (e.g., a formatted raw string, a list of chat roles, or a structured JSON payload) at the time of execution.

---

## Rationale

### Clean Architecture Boundary Preservation

The Domain Layer should remain completely unaware of the target provider's serialization and templating requirements. Decoupling the string formatting ensures the domain model remains pure.

---

### Multi-Provider Independence

The system can target multiple model providers (e.g. local and cloud LLMs) using the same logical `Prompt` entity. Only the compiler in the Infrastructure Layer changes.

---

### Testability

Logical prompt assembly rules (managed by the Grounding Engine) can be verified and tested independently of text serialization rules.

---

### Separation of Concerns

The Grounding Engine is responsible for selecting the context and system instructions. The Infrastructure Provider is responsible for translating those decisions into model-specific API requests.

---

## Consequences

### Positive

- Stronger separation of concerns.
- True provider independence for prompts.
- Cleaner domain entities.
- Easier integration of new models with custom chat templates.

### Negative

- Introduces a new interface (`PromptCompiler`) and its corresponding implementations.
- Slight increase in class count.

---

## Alternatives Considered

### Retaining raw_prompt_string inside Prompt Entity

Rejected.

Forces the Grounding Engine to know the formatting requirements of the active provider, creating tight coupling between domain logic and infrastructure details.

---

### Let Generation Engine compile prompt strings directly

Rejected.

Fails to isolate templating logic as a distinct concern, leading to duplicate formatting code across different language model provider implementations.

---

## Implications

The `Prompt` class definition will be updated to remove `_raw_prompt_string` and its corresponding getter.

A `PromptCompiler` interface will be added to the provider/engine abstractions.
