# ADR-003: Adopt a Local-First Libris

## Status

Accepted

---

## Context

The primary purpose of Libris is to assist students in studying from their own academic textbooks.

The platform is expected to process copyrighted academic material uploaded by the user, including textbooks, lecture notes, and reference material.

Several architectural approaches are possible:

- Cloud-hosted retrieval
- Hybrid cloud/local processing
- Fully local execution

The project also emphasizes:

- Privacy
- Academic integrity
- Offline usability
- Reproducibility
- Low operational complexity

These objectives require careful consideration of where documents and generated knowledge should reside.

---

## Decision

The platform shall adopt a **Local-First Architecture**.

Academic documents, generated embeddings, vector indexes, configuration, and application data shall remain on the user's local machine by default.

Cloud services may be integrated in future versions only as optional infrastructure providers.

The default operating model shall not require:

- Internet connectivity
- User authentication
- Cloud storage
- Remote databases

The platform should remain fully functional in an offline environment once all required models and dependencies are available locally.

---

## Rationale

### Privacy

Academic textbooks often contain copyrighted material.

Keeping all processing local minimizes unnecessary exposure of user-owned documents.

---

### Ownership

Users retain complete ownership of:

- Uploaded books
- Generated embeddings
- Knowledge indexes
- Configuration
- Query history

No document processing should occur outside the user's environment without explicit consent.

---

### Offline Availability

Students frequently study in environments with unreliable internet connectivity.

A local-first architecture ensures uninterrupted access to Libris.

---

### Reduced Complexity

Eliminating mandatory cloud infrastructure simplifies deployment.

The MVP avoids:

- Authentication
- Multi-tenancy
- Cloud synchronization
- Remote storage management

allowing development effort to focus on retrieval quality.

---

### Reproducibility

Local execution provides greater control over:

- Models
- Configuration
- Dependencies
- Data

This improves debugging, benchmarking, and evaluation consistency.

---

## Consequences

### Positive

- Improved privacy
- Offline functionality
- Lower infrastructure complexity
- Faster local development
- Reduced operational cost
- Better reproducibility
- Greater user ownership

### Negative

- Local storage requirements increase
- Index generation depends on local hardware
- No automatic synchronization across devices
- Large document collections may consume significant disk space

These trade-offs are acceptable given the educational focus of the platform.

---

## Alternatives Considered

### Cloud-First Platform

Rejected.

Requires continuous internet access, introduces authentication, increases infrastructure complexity, and creates unnecessary privacy concerns for the MVP.

---

### Hybrid Cloud Architecture

Deferred.

Although future versions may optionally synchronize indexes or settings, introducing hybrid infrastructure during the MVP would significantly increase implementation complexity.

---

### Browser-Only Architecture

Rejected.

Current browser capabilities are insufficient for reliable local embedding generation and scalable vector indexing across large textbook collections.

---

## Implications

Every major subsystem should assume local execution by default.

This includes:

- Document Engine
- Chunking Engine
- Embedding Engine
- Indexing Engine
- Retrieval Engine
- Grounding Engine
- Generation Engine
- Citation Engine

Infrastructure components should expose interfaces that permit future cloud implementations without requiring modifications to domain logic.

Cloud integration, if introduced, should remain an implementation detail rather than an architectural requirement.

## Notes

Local-first should be interpreted as the platform's default operating philosophy rather than a limitation.

Future versions may introduce optional cloud capabilities where they provide clear user value.

Such capabilities must preserve user ownership, transparency, and the ability to operate independently in a fully local environment.
