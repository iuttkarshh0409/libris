# ADR-006: Adopt Typed Prompt Objects

## Status

Accepted

---

## Context

The Libris relies on language models to transform retrieved textbook evidence into coherent academic responses.

Traditional prompt engineering often represents prompts as unstructured strings constructed dynamically within application code.

This approach introduces several challenges:

- Difficult debugging
- Poor version control
- Hidden prompt mutations
- Limited observability
- Tight coupling between retrieval and generation
- Difficult evaluation
- Difficult testing

As the platform evolves, prompt construction will become increasingly sophisticated, supporting multiple interaction modes, retrieval strategies, and language models.

Prompt construction therefore requires the same architectural discipline applied to other domain concepts.

---

## Decision

The platform shall represent prompts as **typed domain objects** rather than raw strings.

Prompt construction shall be the exclusive responsibility of the Grounding Engine.

The Generation Engine shall receive only fully constructed Prompt objects.

Prompt serialization into language-model-specific text shall occur immediately before model invocation.

Prompt objects shall remain immutable after construction.

---

## Rationale

### Separation of Responsibilities

The Grounding Engine prepares evidence.

The Generation Engine communicates with the language model.

Prompt construction belongs to grounding rather than generation.

---

### Improved Testability

Prompt objects can be inspected directly.

Individual prompt sections can be verified without executing a language model.

This significantly improves unit testing.

---

### Better Observability

Structured prompt objects enable:

- logging
- debugging
- inspection
- prompt comparison
- version tracking

without relying on string parsing.

---

### Versioning

Prompt structures evolve over time.

Treating prompts as structured objects enables explicit version management.

Prompt revisions become traceable architectural changes rather than hidden implementation details.

---

### Provider Independence

Different language model providers often require different prompt formats.

The Prompt object remains provider-independent.

Only the serialization layer varies between providers.

---

## Prompt Structure

Every Prompt object should contain well-defined sections.

Examples include:

- System Instructions
- Task Definition
- Retrieved Context
- User Question
- Constraints
- Output Specification

Additional sections may be introduced without affecting existing components.

---

## Consequences

### Positive

- Strong separation of concerns
- Easier debugging
- Better logging
- Deterministic prompt construction
- Improved testing
- Provider independence
- Easier prompt versioning
- Cleaner architecture

### Negative

- Additional implementation complexity
- More domain models
- Serialization layer required

These trade-offs are acceptable given the long-term benefits.

---

## Alternatives Considered

### Raw Prompt Strings

Rejected.

Prompt construction becomes scattered throughout the codebase.

Testing and debugging become unnecessarily difficult.

---

### Template Files

Rejected.

Although preferable to inline strings, template files still lack type safety and explicit structure.

---

### Provider-Specific Prompt Builders

Rejected.

Couples business logic to a specific language model implementation.

Violates Clean Architecture dependency rules.

---

## Implications

The Grounding Engine owns all Prompt object construction.

The Generation Engine must never modify Prompt objects.

Prompt serialization should occur only within Infrastructure adapters responsible for communicating with language models.

Prompt objects should remain independent of:

- OpenAI
- Ollama
- Gemini
- Claude
- future providers

Provider-specific formatting should remain an implementation detail.

## Notes

Prompt objects are considered first-class domain entities within Libris.

The objective is to treat prompt construction as an explicit engineering activity rather than an implementation convenience.

This decision improves maintainability, observability, evaluation, and long-term architectural consistency.
